# 非专业用户的上下文管理方案

你提出了一个非常关键的问题。确实，之前的方法论假设用户具备一定的技术背景。让我为你设计一个**更适合非专业用户**的协作模式。

---

## 核心思路：让 AI 承担技术决策，用户专注于"验收"

```统模式：用户定义规范 → AI 执行 → 用户验收
新模式：  用户描述需求 → AI 定义规范并执行 → 用户验收 → AI 记录
```

---

## 简化版上下文管理（3 层变 2 层）

### 📁 文档结构

```
project/
├── docs/
│   ├── README.md          # 项目说明（AI 生成，用户确认）
│   └── dev-log.md         # 开发日志（AI 自动维护）
└── src/
    └── ...
```

### 📄 README.md（用户能看懂的版本）

```markdown
# 本地图片浏览器

## 这个软件能做什么
- 打开电脑上的图片文件或文件夹
- 查看图片，可以放大缩小
- 可以旋转图片
- 用键盘快捷键操作

## 目前的状态
✅ 已完成，可以正常使用

## 怎么运行
1. 双击 index.html（如果不行，用下面的方法）
2. 或者：在文件夹里打开命令行，输入 `python -m http.server`
3. 打开浏览器访问 http://localhost:8000

## 操作说明
| 操作 | 方法 |
|------|------|
| 打开图片 | 点击"打开文件"按钮 |
| 下一张 | 点击按钮 或 按 → 键 |
| 放大 | 滚动鼠标滚轮 |
| 旋转 | 按 R 键 |
```

## 新的协作流程

### 第一步：用户用自然语言描述需求

**你只需要说：**
> "我想做一个看图软件，能打开图片，能放大缩小，能切换上一张下一张"

**不需要说：**
> ~~"使用 Canvas 2D API，实现 transform 矩阵变换..."~~

---

### 第二步：AI 输出"确认清单"

AI 会把你的需求翻译成具体功能，让你确认：

```markdown
## 我理解的需求，请确认：

### 基本功能
- [ ] 打开单个图片文件
- [ ] 打开文件夹，显示里面所有图片
- [ ] 显示图片
- [ ] 切换到上一张/下一张

### 查看功能
- [ ] 放大/缩小（用鼠标滚轮）
- [ ] 拖动图片位置
- [ ] 旋转图片

### 界面
- [ ] 顶部有操作按钮
- [ ] 底部显示文件名和图片信息

请回复：
1. "确认" - 如果以上都对
2. "补充：xxx" - 如果还想要其他功能
3. "不要：xxx" - 如果某个功能不需要
```

**你只需要回复：** "确认" 或 "补充：还想要幻灯片自动播放"

---

### 第三步：AI 制定技术方案并告知

AI 会简要说明技术方案（你不需要完全理解）：

```markdown
## 技术方案（供参考，不需要你决定）

我会这样实现：
- 用网页技术（HTML + JavaScript）
- 图片显示用 Canvas（一种画图技术）
- 不需要安装任何软件，浏览器直接打开

预计分 4 轮完成：
1. 第 1 轮：能打开和显示图片
2. 第 2 轮：能放大缩小和拖动
3. 第 3 轮：能切换图片
4. 第 4 轮：美化界面

每轮完成后我会让你测试，告诉我哪里有问题。以开始吗？
```

---

### 第四步：每轮结束的验收模板

AI 提供简单的测试清单：

```markdown
## 第 1 轮完成，请测试：

### 测试步骤
1. 用浏览器打开 index.html
2. 点击"打开文件"按钮
3. 选择一张图片

### 预期结果
- [ ] 图片显示在屏幕中央

### 请告诉我
- 如果正常：回复"通过"
- 如果有问题：描述你看到了什么，或者截图给我
```

**你只需要回复：** "通过" 或 "图片没显示，屏幕是空白的"

---

### 第五步：问题排查（AI 主导）

当出现问题时，AI 会引导你：

```markdown
## 需要你帮我检查按 F12 打开开发者工具，点击 Console（控制台）标签，
把红色的错误信息截图发给我。

或者直接把整个屏幕截图发给我，我来分析。
```

## 实际对话示例

### 开始新项目

```户：我想做一个看图软件

AI：好的！我来帮你做。先确认一下需求：
    - 能打开图片文件 ✓
    - 能放大缩小 ✓
    - 能切换上下张 ✓
    - 能旋转 ✓
    还需要其他功能吗？

用户：够了

AI：好的，我开始第 1 轮开发，预计 5 分钟。完成后会给你测试步骤。[生成代码...]
  
    第 1 轮完成！请测试：
    1. 打开 index.html
    2. 点击"打开文件"
    3. 选一张图片看到图片吗？

用户：能看到

AI：很好！第 1 轮通过 ✅
    开始第 2 轮：添加放大缩小功能...
```

### 遇到问题时

```
用户：按钮点不了

AI：我来排查。请告诉我：
    1. 是哪个按钮？（截图最好）
    2. 鼠标放上去是什么样子？（箭头还是禁止符号）

用户：[截图] 是这个下一张按钮，显示禁止符号

AI：我看到了。这是因为还没有打开图片。
    请先点击"打开文件"选择图片，然后再试试"下一张"。
    果还是不行，请按 F12 截图控制台给我看

用户：哦，打开图片后就可以了

AI：好的，这是正常的设计——没有图片时导航按钮会禁用。
    我在文档里补充说明一下。
```

## 给 AI 的提示词模板

如果你开始新项目，可以用这个提示词：

```markdown是一个不太懂编程的用户，想做一个 [描述你的软件]。

请用以下方式帮助我：

1. 先确认需求，用简单的清单让我勾选
2. 技术方案你来决定，简单告诉我就行
3. 分多轮开发，每轮给我简单的测试步骤
4. 遇到问题时，引导我提供截图或错误信息
5. 自动维护一个开发日志，记录做了什么、遇到什么问题

我的需求是：[用自己的话描述]
```

## 总结

| 原来的方式 | 新的方式 |
|------------|----------|
| 用户定义技术规范 | AI 定义，用户确认 |
| 用户写 L1/L2/L3 文档 | AI 自动维护日志 |
| 用户指定接口和命名 | AI 决定并保持一致 |
| 用户排查代码错误 | AI 引导用户提供信息 |
| 用户需要懂技术 | 用户只需要会"测试"和"描述现象" |

**你的核心任务只有两个：**
1. **描述你想要什么**（用日常语言）
2. **测试并反馈结果**（能用/不能用/哪里不对）

技术细节、命名规范、架构设计——这些都交给 AI 来处理和记录。
